2.6 开发环境与生产环境

2.6.1 什么是开发环境与生产环境

环境，就是指项目运行的地方，当项目处于开发阶段，项目运行在开发人员的电脑上，项目所处的环境就是开发环境。当项目开发完成以后，要将项目放到真实的网站服务器电脑中运行，项目所处的环境就是生产环境

2.6.2 为什么要区分开发环境与生产环境

因为在不同的环境中，项目的配置是不一样的，需要在项目代码中判断当前项目运行的环境，根据不同的环境应用不同的项目配置

2.6.3 如何区分开发环境和生产环境

通过电脑操作系统中的系统环境变量区分当前是开发环境还是生产环境

  • 设置系统环境变量

    “此电脑” -> “属性” -> “高级系统设置” -> “高级” -> “环境变量”

  • 在代码中获取系统环境变量的值并对其进行判断，prosess是全局对象下的属性，可以直接调用，process下的属性env代表着运行环境相关的信息

2.7 第三方模块 config

2.7.1 作用

允许开发人员将不同运行环境下的应用配置信息抽离到单独的文件中，模块内部自动判断当前应用的运行环境，并读取对应的配置信息，极大减少应用配置信息的维护成本，避免了当运行环境重复的多次切换时，手动到项目代码中修改配置信息

2.7.2 使用

  • 使用npm install config命令下载模块

  • 在项目的根目录下新建config文件夹（文件夹名字是模块内部规定好的，不可以改为其它名字）

  • 在config文件夹下面新建default.json、development.json、production.json文件

    ps：default.json存放默认的配置信息，development.json存放开发环境的配置信息、production存放生产环境的配置信息。如果当前是开发环境，会读取development.json 文件中的信息；如果是生产环境，会读取production.json文件中的信息；如果要读取的信息在对应的json文件中没有找到，会到default.json文件中查找。

    注意：由于json文件是严格的json格式，对象中的属性必须加双引号

  • 在项目中通过require方法，将模块进行导入

  • 使用模块内部提供的get方法获取配置信息

2.7.3 将敏感配置信息存储在环境变量中

  • 在config文件夹中建立custom-environment-variables.json文件

  • 配置项属性的值填写系统环境变量的名字

  • 项目运行时config模块查找系统环境变量，并读取其值作为当前配置项属于的值

3.知识点

3.1模板文件外联路径

在模板文件中的外联资源的相对路径是相对于浏览器中的请求路径，当请路径改变的时候，模板文件中的外联文件容易找不到，所以使用绝对路径（public为根目录，因为在时渲染模板的时候已经设置了静态资源文件的绝对路径在public下）

3.2 抽离代码

  • 因为许多页面中的头部和侧边栏样式和布局很相似，所以为了减少代码冗余以及方便以后的维护，将头部和侧边栏我的代码抽离到/views/admin/common中

  • 为什么字模板的路径是相对路径？

    因为字模板的是由模板引擎解析的而不是由浏览器解析的，所以字模板的相对路径就是当前路径

3.4 bcrypt加密

在数据库中以明文的方式存储密码是不安全的（如果黑客进入自己的数据库，数据库中的密码还是明文，很容易被盗取用户密码）

哈希加密是单程加密方式（只能加密不能解密）：1234 => abcd

虽然不能解密但是黑客仍然可以通过：将常用的密码经过加密之后存储到数据库中，在存储加密密码的同时也存储明文密码。黑客在入侵数据库后，会将数据库中的加密密码与黑客数据库中的加密密码进行比对，这种破解密码的方式叫暴力破解

如何增加暴力破解的难度？
不仅要对密码加密，还要在加密密码中加入随机字符串可以增加密码被破解的难度

genSalt()接收一个数值作为参数。数值越大随机字符串复杂度越高 ，数值越小复杂度越低，默认值是10。

3.6 cookie与session

  1. cookie介绍

    cookie：浏览器在电脑硬盘中开辟的空间，主要是提供服务器端存储数据。

    （1）cookie中的数据是以域名的形式进行区分的

    （2）cookie中的数据是有过期时间的，超过时间数据会被浏览器自动删除。（在存储数据的时候可以设定存储时间，如果在存储cookie的时候不设置存储时间，浏览器在关闭的时候cookie就会被删除）

    （3）cookie中的数据会随着请求被自动发送到服务器端。ps：在客户端第一次访问服务器端的时候，客户端并没有携带cookie，在服务器响应给客户端后当客户端再次访问服务器端时才会携带cookie（浏览器中的NetWork可以查看）

  2. 在浏览器中查看cookie

  3. session介绍

    session实际上就是一个对象，存储在服务器端的内存中，在session对象中也可以存储多条数据，每一条数据都有一个sessionid作为唯一标识

  4. cookie和session的实现过程

    客户端向服务器端提交邮件地址与密码，服务器端进行验证，在验证成功后会生成sessionid，并将sessionid存储在客户端的cookie中存储在客户端。等到客户端再次访问时，服务器端会从session中匹配客户端携带的cookie中的sessionid，若匹配成功，则响应给客户端相应的数据。

3.9 登录拦截

在显示用户登录后的信息（昵称等）时，如果用户没有登录，userInfo就为空，程序会报错，所以在展示用户的登录信息的时候需要先判断该userInfo是否为空

思路：当客户端向服务器端发送请求的时候，判断请求路径是否是以“/admin”开头的（去了“/admin/login”），如果是就判断该用户是否已经登录，如果已经登录就让程序继续执行；如果没有登录，就重定向到登录页面

ps：app.use(“/admin”)并不是非得匹配“/admin”，而是可以匹配所有以“/admin”开头的路径
